# 数据库迁移功能开发文档 V1

## 1. 背景和目标

Perigon.PostgreSQL 需要提供一组轻量级数据库迁移能力，用来解决两个核心场景：

1. 从已有 PostgreSQL 数据库反向生成 `DbContext`、实体类和映射配置。
2. 根据当前项目中的 `DbContext`、实体类、特性和 Fluent API 配置，在新的 PostgreSQL 数据库中初始创建 schema 和表结构。

本功能只实现“初始创建库和表结构”，不实现 EF Core 完整的增量脚本迁移能力。也就是说，本功能不负责比较两个模型版本之间的差异，不生成 `ALTER TABLE` 升级脚本，不维护 migration history 表，不执行生产数据库结构升级。

设计目标：

- PostgreSQL only：只面向 PostgreSQL，DDL 使用 PostgreSQL 语法。
- NativeAOT safe：类库运行时不扫描程序集，不依赖动态代理、`Reflection.Emit` 或 `Expression.Compile()`。
- 工具灵活性：`Perigon.PostgreSQL.Tools` 是独立命令行工具，不要求 NativeAOT 兼容，可以使用更适合脚手架生成的反射、Roslyn 和模板能力。
- 确定性 SQL：同一模型配置应生成稳定、可预测的 DDL。
- 初始创建优先：支持空数据库或空 schema 的首次创建。
- 映射可回放：从已有数据库生成的代码，应能通过 `EnsureCreated` 在新数据库中创建等价的基础结构。

## 2. 范围

### 2.1 第一版必须支持

- 新增 `DbContext.EnsureCreatedAsync(CancellationToken cancellationToken = default)`。
- 可选新增同步包装 `DbContext.EnsureCreated()`。
- 根据当前 `DbContext` 中的实体模型创建 schema 和表。
- 支持表名、schema、列名、主键、自增列、可空性和数据库类型。
- 支持 PostgreSQL 常见类型映射，包括基础标量类型、数组和 `jsonb`。
- 支持普通表的主键、单列外键和索引元数据。
- 支持确定性默认约束名称。
- 优先识别 .NET 标准 DataAnnotations / Schema 特性；语义与 EF Core 相同的特性不重复定义。
- 新增 `Perigon.PostgreSQL.Tools` 命令行项目，用于从已有数据库生成实体类和 `DbContext`。
- 反向生成工具支持普通表和视图。
- 反向生成工具能识别主键、单列外键、简单唯一索引和普通索引，并尽可能转换为特性配置；复杂对象保持跳过或后续扩展。

### 2.2 暂不支持内容

- 不支持增量迁移脚本生成。
- 不支持 migration history 表。
- 不支持模型版本比较。
- 不支持自动执行 `ALTER TABLE`、`DROP COLUMN`、`RENAME COLUMN` 等结构升级。
- 不支持数据迁移。
- 不支持跨数据库方言。
- 不支持运行时扫描所有程序集查找实体类型。
- 不支持 generated/computed column 的建表 SQL 表达式；当前只识别并明确拒绝 `EnsureCreated` 建表。
- 不保证从复杂数据库对象反向生成完全等价的业务模型，例如触发器、存储过程、自定义 domain、复杂 expression index。

### 2.3 更完善的支持

- 对复合主键、复合外键提供完整建表和反向生成支持。
- 实体与表的继承映射，非抽象类与表一一对应，不支持 TPH/TPT/TPC。
- 对视图生成只读实体和只读查询入口。
- 对 PostgreSQL enum、domain、range 类型提供专门映射。
- 对默认值、长度、精度、检查约束提供更完整的元数据模型。

## 3. 用户场景

### 3.1 从已有数据库生成代码

用户已有 PostgreSQL 数据库，希望生成 Perigon.PostgreSQL 可用的实体类和 `DbContext`：

```powershell
dotnet perigon dbcontext scaffold `
	--connection "Host=localhost;Database=app;Username=postgres;Password=postgres" `
	--context AppDbContext `
	--namespace MyApp.Data `
	--output .\Data `
	--schema public
```

输出内容：

- `AppDbContext.cs`
- 每张表对应的实体类。
- 视图对应的只读实体类。
- 必要的 DataAnnotations / Schema 特性，例如 `[Table]`、`[Column]`、`[Key]`、`[ForeignKey]`、`[NotMapped]`。
- 当前 MVP 主要生成 DataAnnotations / Schema 特性；复杂对象的 Fluent API 配置属于后续能力。

### 3.2 从代码初始创建数据库结构

用户已经定义 `DbContext` 和实体类，希望在新数据库中创建 schema 和表：

```csharp
await using var db = new AppDbContext(options);
await db.EnsureCreatedAsync();
```

预期行为：

- 如果 schema 不存在，则创建 schema。
- 如果表不存在，则创建表。
- 如果表已存在，则不重复创建。
- 对于已存在但结构不一致的表，第一版不做自动修复，应抛出清晰异常或保留 PostgreSQL 原始错误。

## 4. 总体架构

功能拆分为两个方向：

```text
已有 PostgreSQL 数据库
  -> Perigon.PostgreSQL.Tools
  -> Catalog Reader
  -> Database Model
  -> Code Generator
  -> DbContext + Entity + Mapping

DbContext + Entity + Mapping
  -> Source Generator
  -> EntityModel / ColumnModel / RelationshipModel
  -> EnsureCreatedAsync
  -> DDL SQL Builder
  -> PostgreSQL
```

核心组件职责：

| 组件 | 职责 |
|------|------|
| `DbContext.EnsureCreatedAsync` | 运行时入口，执行初始 schema/table 创建 |
| `EntityModel` | 表级元数据，包括 schema、表名、列、主键、外键、索引 |
| `ColumnModel` | 列级元数据，包括列名、CLR 类型、数据库类型、可空性、自增、生成列 |
| `ForeignKeyModel` | 外键元数据，包括依赖表、依赖列、主表、主键列、删除行为和约束名 |
| `IndexModel` | 索引元数据，包括索引列、唯一性和索引名 |
| `CreateTableSqlBuilder` | 根据模型生成 PostgreSQL DDL |
| Source Generator | 从 `DbSet<T>`、特性和 Fluent API 生成 AOT-safe 元数据 |
| `Perigon.PostgreSQL.Tools` | 从已有 PostgreSQL 数据库反向生成代码 |
| Catalog Reader | 读取 `information_schema` 和 `pg_catalog` |
| Code Generator | 将数据库模型转换为 C# 代码 |

## 5. `EnsureCreated` 设计

### 5.1 API

```csharp
public abstract class DbContext : IAsyncDisposable
{
	public Task EnsureCreatedAsync(CancellationToken cancellationToken = default);

	public void EnsureCreated();
}
```

同步方法可以作为异步方法的包装。核心实现应放在 `EnsureCreatedAsync` 中。

### 5.2 执行流程

1. 获取当前 `DbContext` 对应的实体模型列表。
2. 过滤 `[NotMapped]` 类型、非映射属性、private 成员和视图模型。
3. 根据实体模型收集所有需要创建的 schema。
4. 生成并执行 `CREATE SCHEMA IF NOT EXISTS`。
5. 根据外键依赖进行拓扑排序。
6. 生成并执行 `CREATE TABLE IF NOT EXISTS`。
7. 创建表内或表后的外键约束。
8. 创建唯一索引和普通索引。
9. 如果执行过程中出现 PostgreSQL 错误，保留原始异常信息，并附加当前正在执行的实体或表名。

### 5.3 幂等性规则

- schema 使用 `CREATE SCHEMA IF NOT EXISTS`。
- 表使用 `CREATE TABLE IF NOT EXISTS`。
- 索引使用 `CREATE INDEX IF NOT EXISTS` 或 `CREATE UNIQUE INDEX IF NOT EXISTS`。
- 第一版不主动检查已存在表的列定义是否一致。
- 第一版不尝试修改已存在表。

### 5.4 事务策略

`EnsureCreatedAsync` 默认在单个事务中执行当前上下文的初始建表语句。

如果 PostgreSQL 对某些 DDL 或扩展行为有事务限制，后续可以增加选项：

```csharp
public sealed class EnsureCreatedOptions
{
	public bool UseTransaction { get; init; } = true;
}
```

第一版可先不暴露 options API，保持入口简单。

## 6. 元数据设计

### 6.1 `EntityModel` 扩展

现有实体模型需要扩展为能描述建表所需的结构信息：

```csharp
public sealed class EntityModel
{
	public string Schema { get; }
	public string TableName { get; }
	public IReadOnlyList<ColumnModel> Columns { get; }
	public IReadOnlyList<ColumnModel> PrimaryKey { get; }
	public IReadOnlyList<ForeignKeyModel> ForeignKeys { get; }
	public IReadOnlyList<IndexModel> Indexes { get; }
	public bool IsView { get; }
}
```

### 6.2 `ColumnModel` 扩展

列模型需要包含 DDL 生成所需信息：

```csharp
public sealed class ColumnModel
{
	public string PropertyName { get; }
	public string ColumnName { get; }
	public Type ClrType { get; }
	public string? TypeName { get; }
	public bool IsPrimaryKey { get; }
	public bool IsIdentity { get; }
	public bool IsGenerated { get; }
	public bool IsArray { get; }
	public bool IsNullable { get; }
	public int? MaxLength { get; }
	public int? Precision { get; }
	public int? Scale { get; }
	public string? DefaultSql { get; }
}
```

第一版必须支持 `IsNullable`，否则无法可靠生成 `NOT NULL`。

### 6.3 外键模型

```csharp
public sealed class ForeignKeyModel
{
	public string ConstraintName { get; }
	public EntityModel DependentEntity { get; }
	public IReadOnlyList<ColumnModel> DependentColumns { get; }
	public EntityModel PrincipalEntity { get; }
	public IReadOnlyList<ColumnModel> PrincipalColumns { get; }
	public ReferentialAction OnDelete { get; }
}
```

第一版支持：

- 单列外键。
- 显式配置外键。
- 简单约定识别：`UserId` 指向 `User.Id`，列名为 `user_id`。

后续支持：

- 复合外键。
- 自引用外键。
- 更复杂的导航属性推断。

### 6.4 索引模型

```csharp
public sealed class IndexModel
{
	public string IndexName { get; }
	public IReadOnlyList<ColumnModel> Columns { get; }
	public bool IsUnique { get; }
}
```

第一版支持普通 btree 索引和唯一索引。复杂索引作为后续能力处理，例如表达式索引、部分索引、GIN/GiST、排序方向和 INCLUDE 列。

## 7. 模型配置能力

### 7.1 特性配置

因为 Perigon.PostgreSQL 的模型配置体验与 EF Core 相似，所以语义相同的特性应优先复用现有标准库或事实标准库，不在 `Perigon.PostgreSQL` 中重复定义同名特性。

优先级建议：

1. 优先支持 `System.ComponentModel.DataAnnotations` 和 `System.ComponentModel.DataAnnotations.Schema`。
2. 对 DataAnnotations 没有等价物、但社区已熟悉的建模语义，Perigon 应在自身包中提供同语义特性；如果用户历史代码已使用 EF Core 同名特性，Source Generator 可以继续按完整元数据名称兼容识别，但运行时包不应强制依赖 EF Core。
3. 只有 PostgreSQL 特有能力、或标准特性无法表达的能力，才新增 `Perigon.PostgreSQL.Attributes` 特性。
4. 复杂配置优先使用 Fluent API，而不是不断增加特性数量。

推荐直接支持的标准特性：

| 特性 | 命名空间 | Perigon 语义 |
|------|----------|--------------|
| `[Table]` | `System.ComponentModel.DataAnnotations.Schema` | 指定表名和 schema |
| `[Column]` | `System.ComponentModel.DataAnnotations.Schema` | 指定列名、数据库类型和列顺序 |
| `[Key]` | `System.ComponentModel.DataAnnotations` | 指定主键 |
| `[ForeignKey]` | `System.ComponentModel.DataAnnotations.Schema` | 指定外键属性或导航属性，语义与 EF Core 保持一致 |
| `[NotMapped]` | `System.ComponentModel.DataAnnotations.Schema` | 排除类型或属性映射 |
| `[DatabaseGenerated]` | `System.ComponentModel.DataAnnotations.Schema` | 指定 identity、computed 或 none |
| `[Required]` | `System.ComponentModel.DataAnnotations` | 指定非空列 |
| `[MaxLength]` | `System.ComponentModel.DataAnnotations` | 指定字符串、数组或二进制列长度 |
| `[StringLength]` | `System.ComponentModel.DataAnnotations` | 指定字符串最大长度 |

Perigon 内置的扩展特性：

| 特性 | 命名空间 | 处理策略 |
|------|----------|----------|
| `[Index]` | `Perigon.PostgreSQL.Attributes` | 包内置索引特性；Source Generator、运行时 fallback 和 scaffold 代码生成默认使用该特性，并兼容识别已有 EF Core `[Index]` |
| `[Precision]` | `Perigon.PostgreSQL.Attributes` | 包内置精度特性；当前 `EnsureCreated` 已用于 `numeric` 和 `timestamp with time zone` 的 DDL 精度配置 |
| `[Comment]` | `Perigon.PostgreSQL.Attributes` | 包内置注释特性；当前 `EnsureCreated` 会生成 `COMMENT ON TABLE` 和 `COMMENT ON COLUMN` |
| `[Unicode]` | `Perigon.PostgreSQL.Attributes` | 包内置 Unicode 语义特性；PostgreSQL 下当前不改变 DDL 类型，但可作为统一建模元数据被 Source Generator、运行时 fallback 和 Fluent API 接收 |

Perigon 自定义特性应控制在少量 PostgreSQL 或 Perigon 特有能力中；已有 DataAnnotations 或 EF Core 特性可表达的语义优先复用，不重复定义。例如：

| 特性 | 用途 | 原因 |
|------|------|------|
| `[View]` | 标记实体映射到视图 | DataAnnotations 没有标准视图特性 |
| `[DefaultSql]` | 指定数据库默认表达式 | DataAnnotations 没有 SQL 默认值特性 |
| `[GeneratedColumnSql]` | 指定 generated column 表达式 | PostgreSQL generated column 需要数据库表达式 |

示例：

```csharp
using System.ComponentModel.DataAnnotations;
using System.ComponentModel.DataAnnotations.Schema;

[Table("users", Schema = "public")]
public sealed class User
{
	[Key]
	[DatabaseGenerated(DatabaseGeneratedOption.Identity)]
	[Column("id")]
	public int Id { get; set; }

	[Required]
	[MaxLength(200)]
	[Column("email", TypeName = "text")]
	public string Email { get; set; } = "";

	[NotMapped]
	public string RuntimeOnly { get; set; } = "";
}
```

外键示例：

```csharp
public sealed class Blog
{
	[Key]
	public int Id { get; set; }

	[ForeignKey(nameof(User))]
	public int UserId { get; set; }

	public User? User { get; set; }
}
```

如果实体不希望包含导航属性，外键关系应优先通过 Fluent API 配置。

### 7.2 Fluent API 配置

当前已实现一套面向 `EnsureCreated` 的轻量 Fluent API，用于覆盖最常见的表、列、注释、精度、索引和单列关系配置。当前可用形态：

```csharp
protected override void OnModelCreating(ModelBuilder modelBuilder)
{
	modelBuilder.Entity<User>(entity =>
	{
		entity.ToTable("users", "public");
		entity.HasComment("users table");
		entity.Property(x => x.Email)
			.HasColumnName("email")
			.HasColumnType("text")
			.IsRequired()
			.HasComment("user email");
		entity.Property(x => x.Amount).HasPrecision(12, 2);
		entity.HasIndex(x => x.Email).IsUnique().HasDatabaseName("uq_users_email");
	});

	modelBuilder.Entity<Blog>(entity =>
	{
		entity.Property(x => x.UserId).HasColumnName("user_id");
		entity.HasOne(x => x.User)
			.WithMany(x => x.Blogs)
			.HasForeignKey(x => x.UserId)
			.HasConstraintName("fk_blogs_user_id");
	});
}
```

当前已实现的 Fluent API：

- `Entity<TEntity>().ToTable(name, schema)`
- `Entity<TEntity>().HasComment(comment)`
- `Entity<TEntity>().Property(expr).HasColumnName(name)`
- `Entity<TEntity>().Property(expr).HasColumnType(typeName)`
- `Entity<TEntity>().Property(expr).IsRequired()`
- `Entity<TEntity>().Property(expr).HasMaxLength(length)`
- `Entity<TEntity>().Property(expr).HasPrecision(precision, scale?)`
- `Entity<TEntity>().Property(expr).HasComment(comment)`
- `Entity<TEntity>().Property(expr).IsUnicode(isUnicode)`
- `Entity<TEntity>().HasIndex(expr).IsUnique()`
- `Entity<TEntity>().HasIndex(expr).HasDatabaseName(name)`
- `Entity<TEntity>().HasOne(expr)` / `Entity<TEntity>().HasOne<TRelated>()`
- `HasOne(...).WithMany(expr)` / `HasOne(...).WithMany()`
- `HasOne(...).WithMany(...).HasForeignKey(expr)`
- `HasOne(...).WithMany(...).HasConstraintName(name)`

当前尚未实现的 Fluent API：

- `HasKey` / 自定义主键名
- 复合键、复合外键和关系删除行为配置

配置优先级：

1. Fluent API 优先级最高。
2. 特性配置次之。
3. 命名约定最低。

### 7.3 约定规则

默认约定应与现有模型定义保持一致：

| 项目 | 规则 | 示例 |
|------|------|------|
| 表名 | 类名转 snake_case 后复数 | `User` -> `users` |
| 列名 | 属性名转 snake_case | `CreatedAt` -> `created_at` |
| 主键 | `Id` 或 `{TypeName}Id` | `User.Id` |
| 自增 | int/long/short 主键默认 identity | `id integer GENERATED ALWAYS AS IDENTITY` |
| 外键 | `{PrincipalTypeName}Id` | `UserId` -> `users(id)` |
| schema | 默认 `public` | `public.users` |

## 8. DDL SQL 生成规范

### 8.1 标识符

所有 schema、表名、列名、约束名和索引名必须经过统一标识符引用逻辑，生成双引号标识符。

示例：

```sql
CREATE TABLE IF NOT EXISTS "public"."users" (
	"id" integer GENERATED ALWAYS AS IDENTITY,
	"email" text NOT NULL,
	CONSTRAINT "pk_users" PRIMARY KEY ("id")
);
```

### 8.2 类型映射

第一版类型映射：

| C# 类型 | PostgreSQL 类型 |
|---------|-----------------|
| `short` | `smallint` |
| `int` | `integer` |
| `long` | `bigint` |
| `float` | `real` |
| `double` | `double precision` |
| `decimal` | `numeric` |
| `bool` | `boolean` |
| `string` | `text` |
| `DateTime` | `timestamp with time zone` |
| `DateTimeOffset` | `timestamp with time zone` |
| `Guid` | `uuid` |
| `byte[]` | `bytea` |
| `T[]` | 对应元素类型加 `[]` |
| `List<T>` | 对应元素类型加 `[]` |
| `[Column(TypeName = "jsonb")]` | `jsonb` |

如果 `ColumnModel.TypeName` 已显式指定，则优先使用指定类型。

### 8.3 可空性

可空性规则：

- 不可空值类型生成 `NOT NULL`。
- `Nullable<T>` 不生成 `NOT NULL`。
- 启用 nullable reference type 时，`string` 生成 `NOT NULL`，`string?` 可空。
- 源生成器应尽可能读取 Roslyn nullability 信息，避免运行时推断。
- 主键列总是 `NOT NULL`。

### 8.4 约束命名

如果 Fluent API 或特性没有指定名称，则使用确定性默认名称：

| 对象 | 命名规则 | 示例 |
|------|----------|------|
| 主键 | `pk_{table}` | `pk_users` |
| 外键 | `fk_{dependent_table}_{column}` | `fk_blogs_user_id` |
| 唯一约束或唯一索引 | `uq_{table}_{columns}` | `uq_users_email` |
| 普通索引 | `ix_{table}_{columns}` | `ix_blogs_user_id` |

名称过长时应使用稳定截断策略，并附加短哈希，避免超过 PostgreSQL 标识符长度限制。

### 8.5 外键依赖顺序

建表前需要根据外键依赖做拓扑排序：

- 无依赖表先创建。
- 被依赖表先于依赖表创建。
- 自引用外键可以先创建表，再追加约束。
- 循环依赖可以先创建所有表，再使用 `ALTER TABLE ADD CONSTRAINT` 创建外键。

第一版推荐策略：

1. 先创建所有表，只包含列和主键。
2. 再统一创建外键约束。
3. 最后创建索引。

这样可以降低循环依赖处理复杂度。

## 9. Source Generator 设计

`EnsureCreated` 必须满足 NativeAOT 约束，因此当前 `DbContext` 拥有哪些实体类型不能依赖运行时程序集扫描。

Source Generator 需要完成以下工作：

1. 扫描 `DbContext` 子类中的 `DbSet<T>` 属性。
2. 读取实体类型、属性、基类属性和特性配置。
3. 排除 `[NotMapped]` 类型和属性。
4. 排除 private 成员。
5. 生成实体模型注册代码。
6. 为每个 `DbContext` 生成上下文级实体列表。

建议生成形态：

```csharp
partial class AppDbContext
{
	protected override IReadOnlyList<EntityModel> GetEntityModels()
		=> PerigonGeneratedModels.AppDbContextModels;
}
```

运行时 `EnsureCreatedAsync` 只读取 `GetEntityModels()`，不扫描程序集。

### 9.1 NativeAOT 发布约束

类库本身需要支持 NativeAOT，但 Source Generator、Analyzer 和 `Perigon.PostgreSQL.Tools` 不需要支持 NativeAOT。

实现时需要避免将 `PublishAot=true`、`RuntimeIdentifier`、`SelfContained` 等发布属性传递给 `netstandard2.0` 的 Source Generator / Analyzer 项目。否则在执行 NativeAOT 发布时，SDK 会尝试按 AOT 项目处理这些 `netstandard2.0` 项目，并在 restore 阶段报错：

```text
NETSDK1207: 目标框架不支持提前编译。
```

该错误的定义：

- 不是 Perigon.PostgreSQL 运行时代码本身的 AOT 编译错误。
- 不是本机缺少 NativeAOT 工具链的错误。
- 根因是 AOT 发布属性进入了不应 AOT 发布的 Source Generator / Analyzer 项目。

推荐处理方式：

- 类库项目在 `PublishAot=true` 时，不把 Source Generator / Analyzer 作为普通 `ProjectReference` 传入发布 restore graph。
- NativeAOT smoke 项目在发布时先用普通 build 构建 Source Generator，再通过 `Analyzer` DLL 引用它。
- Source Generator / Analyzer 项目继续保持 `netstandard2.0`，不配置 `IsAotCompatible` 或 `PublishAot`。
- `Perigon.PostgreSQL.Tools` 作为 CLI 工具，也不配置 `IsAotCompatible` 或 `PublishAot`。

验收方式：

```powershell
dotnet publish .\tests\Perigon.PostgreSQL.NativeAotSmoke\Perigon.PostgreSQL.NativeAotSmoke.csproj -c Release -r win-x64 /p:PublishAot=true
```

发布成功后应运行生成的可执行文件，确认源生成注册、查询 SQL 生成和基础运行路径在 NativeAOT 下可用。

## 10. 反向生成工具设计

### 10.1 项目

新增项目：

```text
src/
  Perigon.PostgreSQL.Tools/
	Perigon.PostgreSQL.Tools.csproj
	Program.cs
	Commands/
	ReverseEngineering/
	CodeGeneration/
```

工具应发布为 .NET tool，命令名称建议为：

```text
perigon
```

工具项目不需要配置 `IsAotCompatible`，也不需要满足 NativeAOT 发布要求。它可以使用普通 .NET CLI 应用的依赖和实现方式，但生成出的业务代码仍应适配 Perigon.PostgreSQL 类库的 AOT 运行时约束。

### 10.2 命令

第一版命令：

```powershell
perigon dbcontext scaffold `
  --connection "..." `
  --context AppDbContext `
  --namespace MyApp.Data `
  --output .\Data `
  --schema public `
  --force
```

参数：

| 参数 | 必填 | 说明 |
|------|------|------|
| `--connection` | 是 | PostgreSQL 连接字符串 |
| `--context` | 是 | 生成的 DbContext 类名 |
| `--namespace` | 是 | 生成代码命名空间 |
| `--output` | 是 | 输出目录 |
| `--schema` | 否 | 指定 schema，可重复 |
| `--table` | 否 | 指定表或视图，可重复 |
| `--force` | 否 | 覆盖已有文件 |
| `--dry-run` | 否 | 只输出将要生成的文件列表 |
| `--use-fluent-api` | 否 | 尽量使用 Fluent API 而不是特性 |
| `--include-views` | 否 | 生成视图实体，默认开启 |
| `--no-views` | 否 | 不生成视图实体 |

当前 MVP 已实现 `--connection`、`--context`、`--namespace`、`--output`、`--schema`、`--table`、`--force`、`--dry-run` 和 `--no-views`。视图默认包含，可通过 `--no-views` 关闭。`--use-fluent-api` 属于后续能力，当前生成器主要使用 DataAnnotations / Schema 特性表达映射。

### 10.3 Catalog Reader

工具需要读取 PostgreSQL 元数据：

| 信息 | 推荐来源 |
|------|----------|
| schema、表、视图 | `information_schema.tables`、`pg_catalog.pg_class` |
| 列名、类型、可空性、默认值 | `information_schema.columns`、`pg_catalog.pg_attribute` |
| 主键 | `information_schema.table_constraints`、`information_schema.key_column_usage` |
| 外键 | `information_schema.referential_constraints`、`information_schema.key_column_usage` |
| 索引 | `pg_catalog.pg_index`、`pg_catalog.pg_class` |
| 注释 | `pg_catalog.pg_description` |

### 10.4 代码生成规则

生成实体类时：

- 表名转换为 PascalCase 类名。
- 列名转换为 PascalCase 属性名。
- 保留数据库真实名称到 `[Table]` 和 `[Column]`。
- 可空列生成可空 C# 类型。
- 主键、自增、生成列优先写入标准 DataAnnotations 特性，例如 `[Key]`、`[DatabaseGenerated]`、`[Column]`；复杂场景写入 Fluent API。
- 视图默认生成只读实体，且不参与 `EnsureCreated` 建表。
- private 成员不生成。
- `[NotMapped]` 只用于用户手写扩展属性，不从数据库列生成。

生成 `DbContext` 时：

```csharp
public partial class AppDbContext : DbContext
{
	public DbSet<User> Users => Set<User>();
	public DbSet<Blog> Blogs => Set<Blog>();

	protected override void OnModelCreating(ModelBuilder modelBuilder)
	{
		// 复杂关系、复合键、索引和约束名称在这里配置。
	}
}
```

### 10.5 关系识别

反向生成工具需要识别：

- 一对多：外键列位于依赖表，主表主键被引用。
- 一对一：外键列同时具有唯一约束或唯一索引。
- 多对多：中间表只有两个外键，且两个外键组合为主键或唯一约束。

第一版代码生成建议：

- 外键列总是生成普通标量属性，例如 `UserId`。
- 导航属性可以作为可选能力生成。
- 多对多中间表默认仍生成实体类，避免误删业务字段。
- 只有当中间表没有额外业务列时，才可以标记为可折叠的 many-to-many 关系。

## 11. 继承和 `[NotMapped]`

### 11.1 继承

第一版支持读取基类中的 public/protected 可映射属性，并按基类到派生类顺序生成列。

继承映射策略：

- 默认采用单表实体模型，即派生类最终映射到一个表。
- 不自动推断 TPH/TPT/TPC。
- 如果未来支持 TPH/TPT/TPC，需要明确 Fluent API 配置。

### 11.2 `[NotMapped]`

`[NotMapped]` 支持两种使用方式：

- 标记类：整个类型不参与模型注册和建表。
- 标记属性：该属性不参与列映射、查询、写入和建表。

private 成员始终不参与映射。

## 12. 视图支持

反向生成工具应支持从视图生成只读实体：

- 生成 `[Table]` 或后续 `[View]` 标记，保留 schema 和视图名称。
- 视图实体不参与 `EnsureCreated` 的 `CREATE TABLE`。
- 如果视图没有主键，查询仍可支持只读投影，但不支持更新、删除、插入。
- `EnsureCreated` 第一版不负责创建视图定义。

## 15. 错误处理和诊断

`EnsureCreatedAsync` 错误信息应包含：

- 当前实体类型。
- 当前 schema 和表名。
- 当前执行阶段，例如 schema、table、foreign key、index。
- PostgreSQL 原始错误。

反向生成工具错误信息应包含：

- 连接失败原因。
- 无权限读取的对象。
- 无法映射的 PostgreSQL 类型。
- 被跳过的表、视图、索引或约束。
- 文件已存在但未指定 `--force`。

工具遇到不支持对象时，默认输出 warning 并继续生成可用代码；遇到会导致生成代码不正确的对象时，应失败并说明原因。

## 16. 风险和处理策略

| 风险 | 处理策略 |
|------|----------|
| AOT 下无法运行时发现实体 | Source Generator 生成上下文级实体列表 |
| 已存在表结构与模型不一致 | 第一版不自动修复，明确抛错或保留 PostgreSQL 错误 |
| 复杂外键循环 | 先建表，再追加外键约束 |
| PostgreSQL 类型过多 | 第一版覆盖常用类型，未知类型要求显式 `TypeName` 或生成 warning |
| 多对多误判 | 默认保留中间表实体，只在无业务列时提示可折叠 |
| 视图无主键 | 生成只读实体，不参与写入和建表 |
| 标识符过长 | 稳定截断并附加短哈希 |

## 17. 完成定义

第一版完成时，应满足：

1. `DbContext.EnsureCreatedAsync` 可以在空 PostgreSQL 数据库中创建当前上下文的 schema 和表。
2. 基础列类型、主键、自增、可空性、数组、`jsonb` 生成正确。
3. Source Generator 提供 AOT-safe 的上下文实体列表。
4. `Perigon.PostgreSQL.Tools` 可以从已有数据库生成可编译的实体类和 `DbContext`。
5. 生成代码可以在新数据库中通过 `EnsureCreatedAsync` 创建基础结构。
6. 单元测试、集成测试和 NativeAOT smoke 覆盖核心路径。
7. 文档明确说明不支持增量脚本迁移。

## 18. 当前实现状态和限制

截至 `0.1.2`，当前实现已覆盖第一版初始建表的主路径，但仍不是完整迁移系统。

### 18.1 已实现能力

- `DbContext.EnsureCreatedAsync` 和 `DbContext.EnsureCreated`。
- Source Generator 为每个 `DbContext` 注册实体模型列表，避免运行时程序集扫描。
- `CREATE SCHEMA IF NOT EXISTS` 和 `CREATE TABLE IF NOT EXISTS`。
- 基础列、主键、identity、可空性、数组、`jsonb` 和常见 PostgreSQL 类型映射。
- 约定外键：`{PrincipalTypeName}Id` 指向同类型主键。
- 显式 `[ForeignKey]` fallback 识别。
- `Perigon.PostgreSQL.Attributes.IndexAttribute` 已内置到运行时包，同时兼容识别已有 EF Core `[Index]`。
- `Perigon.PostgreSQL.Attributes.PrecisionAttribute`、`CommentAttribute` 和 `UnicodeAttribute` 已内置到运行时包。
- `EnsureCreated` 已支持 `Precision` 到 `numeric` / `timestamp with time zone` 的 DDL 精度映射。
- `EnsureCreated` 已支持表和列注释的 `COMMENT ON` 生成。
- 轻量 Fluent API 已支持表名、表注释、列名、列类型、必填、长度、精度、列注释、Unicode 语义、索引配置和单列外键关系配置。
- 反向生成工具对普通索引和唯一索引输出 Perigon 内置 `[Index]`，不要求用户额外引用 EF Core。
- 普通索引和唯一索引的 DDL 生成与 `EnsureCreated` 执行。
- `[View]` 标记，视图模型不会参与 `EnsureCreated` 建表。
- 非 AOT 的 `Perigon.PostgreSQL.Tools` 项目和 `perigon dbcontext scaffold` MVP。
- NativeAOT 发布路径已避免把 `PublishAot=true` 传递给 Source Generator / Analyzer 项目。

### 18.2 当前限制

- 不支持增量迁移、history 表、模型 diff、结构升级或数据迁移。
- 不支持复合主键、复合外键、表达式索引、部分索引、GIN/GiST、排序方向、INCLUDE 列和检查约束。
- `EnsureCreated` 不校验已存在表结构是否与当前模型一致。
- `EnsureCreated` 暂不创建视图定义。
- 生成列当前缺少 SQL 表达式模型；如果建表模型包含 generated/computed column，`EnsureCreated` 会抛出明确异常，而不是创建成普通列。
- Fluent API 当前只覆盖 `EnsureCreated` 所需的轻量结构配置和单列关系配置，尚未覆盖主键命名、复合关系、关系删除行为和复杂映射。
- 反向生成工具当前读取表、视图、列、主键、单列外键、简单普通索引和唯一索引；默认值、注释、精度、长度、复合外键、INCLUDE 索引和复杂类型仍需后续补齐。
- 反向生成工具已对跨 schema 同名表、列名转换后重复和非法标识符做基础消歧；更完整的复数化、保留字策略和用户可配置命名规则仍属后续能力。
- 类级 `[NotMapped]` 会被 Source Generator 跳过；运行时 fallback 直接拒绝该实体类型。
